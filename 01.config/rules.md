# 🌐 언어 및 커뮤니케이션 규칙 (Language & Communication)

- **최우선 순위:** 모든 응답, 설명, 제안 및 가이드라인은 반드시 **한국어(Korean)**로 작성합니다.
- **No English Responses:** 사용자가 영어로 질문하더라도 답변은 한국어로 수행하며, 전문적인 개발자 동료의 톤을 유지합니다.

- **Primary Language:** 모든 응답, 코드 주석, 설명, 제안 사항은 반드시 **한국어(Korean)**로 작성합니다.
- **No English Responses:** 사용자가 영어로 질문하더라도 한국어로 답변하며, 기술 용어는 관례에 따라 한글(영어 병기) 형태를 권장합니다. (예: "Tracing" -> "트레이싱" 또는 "트레이싱(Tracing)")
- **Technical Terms:** 전문 용어는 업계에서 통용되는 용어를 사용하되, 설명은 명확한 한국어로 제공합니다.
- **Tone & Manner:** 친절하고 전문적인 한국어 개발자 동료의 톤을 유지하세요.
- **주 언어:** 모든 응답, 설명, 가이드라인은 반드시 **한국어(Korean)**로 작성합니다.
- **응답 스타일:** 전문적인 한국어 개발자 동료처럼 답변하며, 기술적 개념을 명확하게 풀어서 설명합니다.

# 💻 코드 및 기술 용어 예외 규칙 (Code & Technical Exceptions)

- **코드 식별자:** 변수명, 함수명, 클래스명, 데이터베이스 컬럼명 등 모든 코드 내 식별자는 **영어(English)**로 작성합니다. (예: `user_name`, `get_interview_results()`)
- **코드 주석:** 코드 내부의 주석(Comments)과 독스트링(Docstrings)은 협업과 가독성을 위해 **한국어**로 작성하는 것을 원칙으로 합니다.
- **기술 용어 병기:** 한국어로 설명하기 어색한 전문 용어는 한글을 주로 사용하되, 괄호 안에 영어를 병기합니다. (예: "비동기 처리(Asynchronous)", "의존성 주입(Dependency Injection)")
- **Commit Message:** 커밋 메시지나 기술 문서의 제목 등은 프로젝트 컨벤션에 따라 한국어 작성을 기본으로 합니다.

# 🛠️ 프로젝트 전용 추가 규칙

- 코드 내의 주석(Docstring 포함)도 한국어로 작성하여 가독성을 높입니다.
- 에러 메시지 분석 시, 에러의 원인과 해결 방법을 한국어로 상세히 풀어서 설명합니다.
- **파일 경로:** 윈도우 공유 폴더 `Z:\`를 컨테이너 내 `/shared_data`로 매핑하여 사용하는 환경임을 항상 인지하고 경로를 제안합니다.
- **Observability:** 모든 LLM 호출 및 파이프라인 로직에는 Arize Phoenix 트레이싱 코드가 포함되어야 합니다.
- **기술 스택:** FastAPI, Celery, Docker, Ollama 환경에 최적화된 Python 3.10 코드를 생성합니다.
